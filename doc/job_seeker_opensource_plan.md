## 2. Must-privatize list (PRIVATE)

These paths must **never** be pushed to the public repo. The top-level `.gitignore` enforces this:

| Path | Why private |
|------|-------------|
| `outputs/*` (except `outputs/README.md`) | Contains compiled PDFs with real contact info, application tracker CSVs, cover notes, job batch notes |
| `history/` | Interview debriefs, application records — personal + potentially employer-sensitive |
| `meessage/` | Telegram export HTML files (note: directory is misspelled "meessage" locally — leave as-is) |
| `doc/jd_mapping/` | Company-specific JD analyses (real company names, real JD text) |
| `doc/*.docx` | Employer IP — system design docs from former/current employers (e.g. `数据交换服务系统V1.6.0项目系统设计说明书.docx`, `北金所开放接口平台-系统设计说明书-V1.0-20221122.docx`) |
| `doc/extracted/` | Markdown extractions from the above `.docx` files |
| `doc/experience_bank.md` | Real bullet material with real metrics — ship `.example.md` instead |
| `LaTeX_Resume_CN/*.tex` (until sanitized) | Contain real name `YOUR_NAME` and real email |
| `LaTeX_Resume_EN/*.tex` (until sanitized) | Contain real name and real email |
| `LaTeX_Resume_CN/*.aux`, `*.log`, `*.out`, `*.pdf` | Build artifacts (regenerated by `build_resumes.sh`) |
| `LaTeX_Resume_EN/*.aux`, `*.log`, `*.out`, `*.pdf` | Build artifacts |

## 3. Sanitization strategy

### 3.1 Experience bank

Create `doc/experience_bank.example.md` with the same structure as the real `experience_bank.md` but with placeholder content:

```markdown
# Experience Bank (Example)

## Role: ROLE_TITLE @ COMPANY_X (YYYY-MM — YYYY-MM)

### Bullet material

- Designed and maintained COMPONENT_Y, processing METRIC_N events/day
- Led TEAM_SIZE engineers through LIFECYCLE_PHASE (design → deploy → operate)
- Built TOOL_Z for PURPOSE, reducing METRIC by PERCENT%

## Honesty boundaries (per role)

- Do not claim: ASPIRATIONAL_CLAIM
- Correct framing: ACTUAL_SCOPE
```

### 3.2 Resume `.tex` files

Two options (see §1):

- **Option A**: Ship `resume_template/` only; document the clone-and-customize flow in README.
- **Option B**: Sanitize the production `.tex` files with a script that replaces real name → `YOUR_NAME`, real email → `your-email@example.com`, etc. Run before every public push. Keep the real `.tex` files in a private branch (`private/main`) and merge only sanitized versions to `main`.

### 3.3 JD mapping examples

If users want to see how JD mappings look, provide `doc/jd_mapping/.example/` with one sanitized example:

```markdown
# JD Mapping: COMPANY_X

## Role
ROLE_TITLE (TEAM)

## Key requirements
- Requirement 1
- Requirement 2

## Resume variant to use
resume-zh_jd_company_x.tex (Plan A: headline-only customization)

## Headline to use
\centerline{ROLE_TITLE — TECH_STACK_FOCUS}
```

### 3.4 Tracker schema

Publish the tracker CSV schema (column names) in `doc/tracker_schema.md` so users can set up their own tracker. Do **not** publish actual tracker rows.

```csv
date,company,role,location,variant_used,jd_source,status,notes
```

## 4. Roadmap (3 phases)

### Phase 1 — Cross-platform build + Skill skeleton (this sprint)

**Goal**: Anyone can clone the repo, install TeX, run `./build_resumes.sh`, and get PDFs. The Skill teaches the workflow.

**Deliverables**:

- `build_resumes.sh` (mirrors `build_resumes.ps1`, adds page-count parsing)
- `doc/BUILD_MAC.md` (macOS TeX install instructions)
- `.workbuddy/skills/job-seeker/` (WorkBuddy-native skill, `agent_created: true`)
- `docs/skill/job-seeker/` (portable mirror for open-sourcing)
- `doc/job_seeker_opensource_plan.md` (this document)
- README update (macOS section, open-source plan link, relative paths)
- Top-level `.gitignore` (privatization)
- `.github/workflows/build-resumes.yml` (CI)

**Acceptance**:

- `./build_resumes.sh` runs end-to-end on macOS, exits 0, produces all 12 PDFs in `outputs/`
- `outputs/resume_job_en.pdf` is 1 page
- Skill loads in WorkBuddy (passes `quick_validate.py`)

### Phase 2 — Format conversion pipeline

**Goal**: Single source document → multiple output formats (Markdown, DOCX, PDF, LaTeX). Reduces manual sync between `.tex` and any Markdown/DOCX mirrors.

**Recommended tools** (in order of adoption):

| Tool | Use case | Why |
|------|----------|-----|
| `pandoc` | MD ↔ LaTeX ↔ DOCX general conversion | Mature, widely available, handles most structure |
| `python-docx` | Programmatic DOCX generation/editing | Fine-grained control when pandoc output isn't perfect |
| `latexmk` | LaTeX → PDF automation (watches for changes) | Already standard in TeX distributions; better than calling xelatex manually |

**Architecture**:

```
convert/
├── __init__.py
├── adapters/
│   ├── base.py                 # Adapter ABC: convert(input_path, output_path) -> bool
│   ├── markdown_to_latex.py    # uses pandoc
│   ├── latex_to_pdf.py         # uses xelatex / latexmk
│   ├── latex_to_docx.py        # uses pandoc
│   ├── docx_to_latex.py        # uses pandoc (lossy)
│   └── markdown_to_pdf.py      # uses pandoc + xelatex
├── pipeline.py                 # chains adapters: e.g. MD → LaTeX → PDF
└── cli.py                      # `python -m convert --input foo.md --output foo.pdf`
```

Each adapter takes `(input_path, output_path)` and returns success/failure. A `Pipeline` class chains adapters. This makes it easy to add new format pairs without touching existing code.

**Phase 2 non-goals**:

- Don't try to make LaTeX ↔ DOCX lossless — some formatting will diverge. Document the limitations.
- Don't auto-rewrite existing `.tex` files from a Markdown source. The `.tex` is the source of truth for PDF; Markdown is for editing convenience.

### Phase 3 — Browser auto-application (human-in-the-loop)

**Goal**: Read a JD URL → pick a resume variant → prefill the application form → **pause for user confirmation** → submit → record to tracker.

**Design principles** (from master prompt §3.3):

1. **Human-in-the-loop**: prefill OK; final "submit" click requires explicit user confirmation. No exceptions.
2. **Site adapters**: one adapter per recruiting site. No monolithic scraper.
3. **Audit log**: every operation writes to `history/` or tracker columns (timestamp, site, PDF used, success/failure).
4. **Anti-crawl & ToS**: respect site terms. On failure, degrade to "open page + prefill fields" (no submit).
5. **Sensitive info**: credentials via env vars or system keychain — never in repo.

**Architecture**:

```
apply/
├── adapters/
│   ├── base.py                 # Adapter ABC: read_jd, prefill_form, submit (submit requires confirmation)
│   ├── boss_zhipin.py          # Boss直聘
│   ├── linkedin.py             # LinkedIn
│   ├── official_career_page.py # Generic ATS (Workday, Greenhouse, Lever)
│   └── ...
├── mapping/
│   ├── jd_to_variant.py        # rule-based: JD keywords → recommended variant
│   └── variant_renderer.py     # loads .tex → compiles PDF → returns path
├── confirm.py                  # human-in-the-loop gate (CLI prompt or GUI dialog)
├── audit.py                    # writes to history/ and tracker
└── cli.py                      # `python -m apply --url https://... --variant jd_company_x`
```

**Tooling choice**:

- **If staying in WorkBuddy**: use the `ClawBrowser` skill (Playwright CLI wrapper). Best when the user wants the agent to drive the browser.
- **If cross-platform / standalone**: use raw Playwright (Python or Node). More setup but no editor dependency.

**Phase 3 non-goals**:

- No bypassing CAPTCHAs or bot detection. If a site blocks automation, fall back to "open page + show prefill data" — user copies manually.
- No credential storage in the repo. Use `keyring` (Python) or system keychain.
- No auto-submit, ever. The `submit()` method always blocks on `confirm()`.

- [ ] `history/`, `meessage/` are empty or absent from the working tree
- [ ] `doc/jd_mapping/`, `doc/*.docx`, `doc/extracted/`, `doc/experience_bank.md` are absent

- [ ] `doc/experience_bank.example.md` exists with placeholder content
- [ ] `doc/tracker_schema.md` exists with column definitions
- [ ] `LICENSE` file added (MIT or Apache-2.0)
- [ ] `README.md` updated with macOS section, open-source plan link, relative paths
- [ ] CI workflow passes on a test push to a private remote first
